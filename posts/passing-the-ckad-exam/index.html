Passing the Certified Kubernetes Application Developer Exam in 2020

Certification Background

The Certified Kubernetes Application Developer (CKAD) exam is one of two leading Kubernetes certification exams provided by the Cloud Native Computing Foundation (CNCF). The exam’s focus is on understanding how to use kubectl and a vanilla Kubernetes installation to quickly spin-up, spin-down, or generally interact with application container architectures. Some examples include creating pods with sidecar containers, understanding how to create most basic API primitives via CLI or YAML manifest, and interacting with multiple namespaces. This exam is time-boxed at 2 hours. The CNCF also provides another certification exam, called the Certified Kubernetes Administrator (CKA), which focuses more on managing/administering the Kubernetes control plane. This exam is time-boxed at 3 hours. Both exams cost 300 USD as of 02/2020, but I used the discount code KUBERNETES15 which I found using some creative reddit searching, bringing the cost of the exam down about 15%.

Why did I go with the CKAD as opposed to the CKA? I chose to pursue the CKAD because I didn’t possess an intimate interest in actually administering Kubernetes (I find a lot of places prefer to use tools like kops for these sorts of things), but rather just wanted to demonstrate a good baseline understanding of working with typical Kubernetes applications and patterns. I also read the pass rate for the CKAD was lower (66% required to pass as of 02/2020), and required less namespace and cluster context-switching (I’ll touch on this later in the post).

Exam Study Outline / Strategy

How did I study for this exam and pass it? Keep in mind, this strategy was useful given to me with only a prior high level understanding of how Kubernetes worked, but no real-world production experience using it, and no day to day workloads or things to keep these concepts at the forefront of my mind.

I broke down the preparation into a few distinct phases:

  1. Setup a practice Kubernetes cluster for subsequent phases
  2. Watch the core tutorial/video content and follow alone with practice environment, focusing on knowledge acquisition
  3. Compile a list of exam preparation quizzes
  4. Practice each quiz, attempting to create or define the Kubernetes resources using speed techniques (such as shorthand kubectl CLI commands and bash aliasing)
  5. Review exam tips prior to the exam

Phase 1: Practice Environment Buildout

Integral to this certification prep strategy is having your own Kubernetes environment to practice along with, both for doing the basic introductary learning exercises from the video content, as well as for working with API primitives for speed preparation examples. Going through all of the ways one could setup a Kubernetes cluster is a bit beyond the scope of this blog post, but I used LinuxAcademy’s Cloud Playground feature to keep 3 Linux VMs spun-up, and installed Kubernetes on those using the instructions provided by the companion course Certified Kubernetes Application Developer by Will Boyd. This LinuxAcademy course was also the resource I used for the core video tutorial content in the next phase.

Why did I choose to create playground environment on the LinuxAcademy platform, as opposed to building out my own on a set of dedicated hosts that I own? The answer is that the LinuxAcademy Cloud Playground offering allows you to access all of the cloud server infrastructure you spin up completely through a browser instance, which was a mandatory requirement for me to be able to use this practice environment within the boundaries of a highly regulated financial company’s network (since the traffic is now just flowing over ports 80 and 443, as opposed to SSH over port 22, which is typically blocked at the firewall level in most secured environments).

Phase 2: Training for Knowledge

As mentioned earlier, I used the Certified Kubernetes Application Developer course on LinuxAcademy by Will Boyd, for learning the fundamentals of both Kubernetes and kubectl, as well as an overview of the CKAD exam, its structure, and a walkthrough of setting up a practice environment using the readily available Cloud Playground feature of the LinuxAcademy platform.

I watched each lesson once, and made sure to follow along with all of the exercises in the lessons on my own practice environment, because I find that hands-on learning always beats just watching someone else do it 9 times out of 10. This course also provides some time-boxed Lab Assignments, which are more rigorous examples that give you real world certification exam-based questions and scenarios, for you to practice for longer periods of time. I did each of these lab assignments in my demo environments twice, but I found that these weren’t rigorous or in enough depth to sufficiently prepare you for the exam on their own. They were great for learning the basic concepts, however.

I highly recommend this course. I digested it in chunks of at most 1 hour per day, but even a moderately aggressive study pace, one could easily get through all of this lecture content in about 10 hours total.

Phase 3: Training for Speed

Once I completed watching the video lectures and tutorials, I asked myself if I had a good conceptual understanding with the following topics:

  • The basic Kuberentes API primitives (pods, deployments, volumes, configmaps, secrets, services, etc)
  • A good handle on how to write basic YAML Kubernetes resource manifests (the LinuxAcademy course above drills this hard, so you should be good to go)
  • Comfortable using the kubectl CLI utility to perform basic operations, like instantiating a manifest file, listing pods, etc.
  • Understanding how Kubernetes targets pods using selector patterns in the metadata
  • A basic understanding of the sidecar container pattern (at this point I couldn’t write a manifest to create one from memory, but could work through it with online help)

At that point, I shifted gears and focus from increasing my understanding of how Kubernetes works and the less utilized API objects, to being able to create all of the previous API objects I learned using only the kubectl CLI utility, as fast as possible.

You may be asking yourself - “Why would you devote half of your study effort to re-learning how to create k8s API objects using kubectl, when you already know how to create them with a manifest file? The answer is - in order to pass the exam, you have to. It’s really as simple as that. I’m the kind of person who prefers to take their time and be sure I know the answer to a question before I answer it, but the CKAD exam seems to focus more on how quickly you can spin up the resources in the desired state, rather than how well you understand how they all fit together to form a solid application architecture.

This is evident by the fact that the exam gives you 19 questions, which varying weights of the overall score, and only two hours to complete the exam. That may seem like a lot of time, but each question is a multi-step problem, and requires you to do the following pre-work before you can even begin answering the question:

  • Ensure you are connected to the right k8s cluster, and if you are not, switch to the right cluster (if you answer a question in the wrong cluster, you get no credit)
  • Ensure you are attached to the right namespace (the exam seems designed to purposely confuse you here. I wasted almost 2 minutes per question making sure I was in the right namespace)
  • Read the question and understand everything it is asking you to create (sometimes it might have 3-4 different steps, including a post-step to validate what you created actually works)

I really want to harp on the namespace thing a bit more. The exam actually used namespace values which contained capital “O” and “0” intermixed in the names of the namespaces (WTF), and each namespace name was a non-intuitive collection of letters and numbers. When each question buries which namespace you need to connect to half-way through the wording, or omits it entirely, alerting you that you need to be connected to the “default” namespace, you will probably find yourself starting to create an object to begin answering a question, then look back at the question and realize you forgot to connect to the right namespace. So then you’ll have to delete whatever you just created, switch to the right namespace, and start over. All of this eats precious time. Because of this variance, you really have to be able to reliably create the actual API objects as fast as possible, so you can spend time doing this error-handling of the test environment.

Speed Training Resources

The following are the three sample question training resources I found through research to adequatly prepare you for the CKAD exam (assuming you have the foundational knowledge):

  1. https://github.com/dgkanatsios/CKAD-exercises
  2. https://github.com/bmuschko/ckad-prep
  3. https://medium.com/bb-tutorials-and-thoughts/practice-enough-with-these-questions-for-the-ckad-exam-2f42d1228552

There are a ton of other great resources online now which have sample questions and practice scenarios to help prepare you for the CKAD exam. However, I limited my training material to the these three URLs, because the sample questions they provided were the most comprehensive, updated, and relevant to the actual CKAD exam format. The only thing I’ll say is that all of these resources lacked sufficient preparation for questions dealing with network policies and ambassador container patterns. Make a note of that for your own study plans.

Ideally, when going through these problem sets, you should be practicing them on your own Kubernetes cluster (or a vendor provided one like LinuxAcademy), trying to do all of them using ONLY the CLI whenever possible. This will teach you which resources can and cannot currently be instantiated with a one-line kubectl create/run command, and only by going through these problem sets multiple times will you gain enough mind-muscle memory to be able to create pods, deployments, and services, instinctively. I’d say to be safe, go through each of these problem sets twice. After each completion of an entire problem set, make a mental note of which k8s API object you are still not super familiar with creating quickly on the command-line, and return to that one again to study specifically when you are mentally recharged.

Phase 5: Other Very Useful Exam Tips

Be Mindful of Varying Question Weights

The exam had a total of 19 questions, but each question ranged from being worth 2% of the overall grade, to up to 15%! Since the questions are unevenly weighted, it makes a ton of sense to spend the first 5 minutes of your exam skimming through all 19 questions of your exam, and making mental notes of which ones are the big-bangers vs. the paltry 2-3% questions. Typically, you’d want to answer all the highest percentage of grade questions first, and save the bulk of your remaining time for the smaller % ones near the end.

The only caveat to this strategy is if, during your skimming of the questions, you notice a low value question which can be answered almost immediately (i.e. it requires no context or cluster switching, can be solved with a single kubectl create command, and requires no post-validation) - in those situations, you should knock those out immediately and net some easy points.

Time-Saving Bash Aliases

As mentioned earlier, speed of completion of each question is the most important factor in passing this exam. At the beginning of my exam (literally the first 3 minutes), I added the following bash aliases to my shell, which greatly sped up the subsequent kubectl calls needed to answer all of the questions:

  • $ k=kubectl
  • $ kx='kubectl explain'
  • $ kcscn='kubectl config set-context --current --namespace'

Having to type k instead of kubectl greatly reduces the potential error-surface of mistypes you can make, in an exam where literally every second counts.

Using ‘kubectl explain’

kx is another useful alias because although the exam lets you reference the online documentation in another browser tab, the internet connection speed and the exam interface become limiting factors in finding answers quickly. I found that it was much faster to lookup API reference help by using the kubectl explain object_name.field1.field2.fieldN... | grep \< pattern to quickly find the proper way to format a YAML document in cases where the vanilla kubectl create command couldn’t instantiate the object directly. I only really used the reference browser tab to look up how to setup an ambassador pod (because I completely blanked on this one - I ended up not even having enough time to complete it).

With the kx alias, that command can be called much quicker, like kx pod.spec.containers. Remember - reducing potential fat-fingers on this exam = good idea.

Be Aware of Copy-Paste Limitations on Mac

If you decide to take the exam on OSX like I did, be very aware that the copy-paste hotkeys may not work as expected in the exam interface. Be prepared to copy-paste blocks of text into the built-in notepad, or use right mouse-clicks to copy-paste text blocks (this becomes important when you are modifying YAML files to add additional containers with almost identical fields - pretty common in the adapter and logging pod questions)
